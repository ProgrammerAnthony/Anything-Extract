# 前端代码质量与组件设计规则

本规则用于审查 React/Next.js + TypeScript 项目中的组件/页面/Hook 代码质量与可维护性。重点关注：

- 组件职责是否清晰
- 类型是否健全
- 逻辑是否易读、易扩展
- 是否便于复用与测试

## 1. TypeScript 类型与接口

- **避免滥用 `any`**：
  - 尽量使用明确的接口/类型别名描述数据结构。
  - API 响应、表单数据、业务实体等应有明确的 TypeScript 类型。
- **为组件 Props 声明类型**：
  - 所有组件 Props 使用 `interface` 或 `type` 定义，而非隐式 `any`。
  - 公共组件的 Props 类型应位于单独文件或集中位置，便于复用。
- **使用更精确的类型**：
  - 对有限集合（如状态、标签类型）使用联合类型或枚举，而不是普通字符串。
  - 布尔/可选值使用显式类型，避免混用 `string | boolean` 等模糊设计。

## 2. 组件拆分与职责

- **页面组件只做页面编排**：
  - 页面 (`app/.../page.tsx`) 负责路由、布局拼装与少量页面级状态。
  - 复杂表单、大段展示区域、列表行、卡片等应拆分为子组件。
- **单一职责原则**：
  - 一个组件/Hook 不应同时负责数据获取、复杂业务计算与庞大 UI。
  - 若文件行数过长、嵌套层级过深或包含多块独立 UI，优先拆分。
- **复用常见模式**：
  - 多处重复出现的卡片、按钮、表格行、过滤条等应抽为可复用组件。
  - 避免「复制粘贴同一结构再微调几个类名」的实现方式。

## 3. Hooks 与副作用

- **正确使用 `useEffect` 依赖**：
  - 保证依赖数组完整，避免遗漏依赖导致状态不同步。
  - 将稳定的回调用 `useCallback` 包装后再作为依赖，避免不必要的重复执行。
- **避免副作用泄漏**：
  - 在 `useEffect` 中注册的事件监听或定时器，必须在 cleanup 中移除。
  - 不在渲染过程中（组件函数体）直接执行副作用（如网络请求、DOM 操作）。
- **逻辑抽取为自定义 Hook**：
  - 重复的「加载数据 + loading/error 状态管理」逻辑抽取为 `useXxx` Hook。
  - 复杂的业务状态机可以封装在 Hook 内，对外暴露清晰的状态与操作。

## 4. 可读性与结构

- **清晰的命名**：
  - 变量、函数、组件、Prop 命名体现业务语义（如 `selectedTagIds`/`currentStage` 等）。
  - 禁止使用 `data1`、`list2`、`infoObj` 之类模糊命名。
- **逻辑分段与注释**：
  - 将「状态声明」「数据加载」「事件处理」「渲染函数」等分段组织，避免交织。
  - 仅在必要处添加注释，解释「为什么这样做」，而非复述代码本身。
- **复杂 JSX 的拆分**：
  - 对于包含多层嵌套的 JSX，应拆成小组件或局部渲染函数，以提高可读性。
  - `return` 中逻辑过多时，抽取部分 UI 为 `renderXxx()` 函数或独立组件。

## 5. 错误处理与边界情况

- **网络错误与异常**：
  - 所有异步请求应处理错误（`try/catch` 或 `.catch`），不能只 `console.error` 然后静默失败。
  - 为用户提供友好的错误提示与重试方式（按钮或刷新提示）。
- **空数据与边界**：
  - 对空列表、未选择项、缺少配置等情况提供明确的提示或引导（如「当前还没有标签」「请先选择知识库」）。
  - 避免因为 `undefined`/`null` 导致运行时崩溃。

## 6. 路由与导航

- **统一的导航方式**：
  - 在 Next.js 中使用官方路由 API（如 `next/link`、`next/navigation`）进行导航，而非手动拼接 URL 并刷新页面。
  - 导航逻辑集中管理在合适的组件或布局中，避免到处散落硬编码路径。
- **路径与参数解析**：
  - 从 URL 中解析参数时应处理不存在/格式不正确的情况。
  - 在依赖路由参数的组件中，检查参数存在性后再发起请求或执行后续逻辑。
