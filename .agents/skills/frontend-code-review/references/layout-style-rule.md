# 布局、样式与 UI 一致性规则

本规则用于审查前端界面的整体布局、样式规范与视觉一致性，尤其适用于使用 Tailwind CSS、全局布局组件和统一设计语言的项目。

## 1. 全局布局与框架

- **复用统一布局**：
  - 页面应通过顶层布局组件（如主布局、侧边栏、页面头部等）实现基本框架，而不是在每个页面重新搭建。
  - 新增页面应复用已有的布局与上下文（如页面头部组件、侧边栏折叠状态），保证交互一致。
- **背景与间距统一**：
  - 页面背景色、主内容区域的 padding、卡片间距应与既有页面保持一致。
  - 避免同一层级的页面使用多种接近但不一致的背景色（如 `#f3f6fd` 与 `#f5f7fa` 混用）。

## 2. Tailwind 与设计令牌

- **减少魔法值**：
  - 避免在各处随意使用十六进制颜色或随机的间距值。
  - 常用主色、次色、灰度与阴影建议抽象为 Tailwind 主题扩展或 CSS 变量。
- **遵循一致的间距与圆角体系**：
  - 按设计系统定义的步进（如 `4/8/12/16px`）选择 `p-x`、`m-x`、`rounded` 等类。
  - 避免在相似组件中使用明显不同的圆角和阴影。
- **可读性优先**：
  - Tailwind 类名尽量按功能分组（布局、尺寸、颜色、排版），避免无序堆叠。
  - 对复杂样式可提取为组件级 className 或组合组件，减少重复拼接。

## 3. 组件与视觉模式复用

- **卡片、按钮、标签等模式的一致性**：
  - 常见 UI 模式（如卡片容器、主要按钮、次要按钮、标签/Chip 等）应有统一的实现方式。
  - 若不同页面上出现的按钮/卡片样式「看起来类似但细节不同」，应考虑复用组件或抽象公共样式。
- **避免内联风格偏差**：
  - 优先使用 Tailwind 类控制样式，避免在 JSX 内混用大量 `style={{ ... }}`。
  - 特殊情况（如动画或第三方库要求）使用内联样式时，应有清晰的理由。

## 4. 响应式与适配

- **检查不同屏幕宽度下的表现**：
  - 使用 `sm` / `md` / `lg` / `xl` 等断点类保证布局在常见屏幕尺寸上都可用。
  - 避免固定宽度/高度导致小屏幕出现横向滚动或内容溢出。
- **栅格与列布局**：
  - 使用 Tailwind 栅格（`grid-cols-*`）或 Flex 组合实现自适应布局。
  - 多列布局在小屏下应自动折行到单列，避免内容过于拥挤。

## 5. 文本、图标与可访问性

- **文字层级与颜色**：
  - 标题、副标题、正文、辅助文案的字号与颜色应遵循统一的层级体系。
  - 避免使用过浅的灰色或低对比度颜色影响可读性。
- **图标使用**：
  - 使用统一的图标库（如某个 React 图标集），保持风格一致。
  - 图标大小应与文本字号协调，避免同一页面内出现大小差异极大的图标。
- **可访问性基础**：
  - 交互元素（按钮、链接）应有清晰的 hover/active 状态。
  - 表单控件应有可见标签或 `aria-label`，图像应提供 `alt` 文本（若是纯装饰图标可以标记为隐藏）。

## 6. 全局样式与局部覆盖

- **全局样式的使用**：
  - 全局样式文件仅用于基础重置、字体与全局滚动条等，不应塞入与单个页面强耦合的样式。
  - 若某业务模块需要较多定制样式，应考虑组件级样式或局部 CSS 模块，而不是改动全局。
- **覆盖顺序与冲突**：
  - 避免通过层层增加 `!important` 来解决样式冲突；优先调整类顺序与结构。
  - 出现样式混乱时，应检查是否同时使用了 Tailwind 类与自定义 CSS 进行相同属性的设置。
