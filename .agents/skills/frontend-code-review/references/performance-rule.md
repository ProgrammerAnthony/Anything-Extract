# 性能与交互体验规则

本规则用于审查前端代码在渲染性能、网络请求与交互体验方面的潜在问题，尤其适用于使用 React/Next.js 的单页应用或多页应用。

## 1. 渲染性能与重渲染控制

- **避免不必要的重渲染**：
  - 在 `.map` 渲染列表时使用稳定且唯一的 `key`，不要使用 `index` 作为长期列表的 key。
  - 避免在 JSX 中频繁创建匿名函数、对象或数组；对于复杂逻辑可提前计算或使用 `useMemo`/`useCallback`。
- **组件粒度合理**：
  - 将大块列表或复杂视图拆分为子组件，减少父组件状态变更对整个树的影响。
  - 对于频繁更新的局部区域，尽量收敛状态到该区域内部。

## 2. 大列表与数据展示

- **分页与懒加载**：
  - 对于可能较大的列表，优先采用服务端分页或前端分页，而不是一次性渲染所有项目。
  - 对附加信息（如检索详情、调试信息）使用可折叠区域或「展开详情」按钮，避免默认渲染过多内容。
- **必要时使用虚拟列表**：
  - 当列表条数较多（数百条以上）且每条渲染成本较高时，可考虑引入虚拟列表方案。

## 3. 网络请求与并发

- **请求合并与去重**：
  - 同一页面中应避免多处独立发起完全相同的请求；可提升到父组件或统一的 Hook 中。
  - 对依赖关系明确的请求（如先加载配置再加载数据）按顺序组织，避免不必要的并行请求失败重试。
- **合理的轮询与刷新**：
  - 若需要轮询状态（如任务进度、索引状态），控制轮询间隔并在完成/失败后停止。
  - 避免在组件卸载后仍继续执行定时器或请求回调。

## 4. 客户端/服务端边界（适用于 Next.js 等框架）

- **划分客户端组件与服务端组件**：
  - 仅在需要浏览器能力（如事件、DOM 操作、状态 Hook）时才将组件标记为客户端组件。
  - 纯展示型、依赖后端数据但无交互状态的部分，优先作为服务端渲染或静态内容。
- **避免在渲染中执行重计算**：
  - 对 CPU 密集型计算或格式化，可使用 `useMemo` 缓存结果，或移动到后端/构建阶段。

## 5. 交互反馈与可用性

- **明确的加载与禁用状态**：
  - 异步操作进行中，相关按钮应显示「处理中」文案并处于禁用状态，避免重复触发。
  - 表单提交、批量操作等应有明显反馈（成功提示、错误消息等）。
- **错误信息的呈现**：
  - 错误提示应与当前操作有关，并引导用户可能的解决方式（检查配置、重试、跳转到相关页面等）。
  - 避免只显示通用「失败」提示，而不说明下一步。

## 6. 资源加载与打包

- **按需加载组件**：
  - 对于仅在特定场景出现的重组件（复杂图表、大型编辑器等），可考虑使用动态引入或懒加载。
- **静态资源与图片**：
  - 使用框架提供的图片组件（如 Next.js 的 `Image`）管理图片大小与懒加载，避免无约束的大图直接加载。
  - 对于纯装饰性图片或图标，可使用矢量图标库，以减少静态资源体积。
