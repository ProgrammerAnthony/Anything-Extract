# 状态管理与数据流规则

本规则用于审查前端代码中的本地状态、全局状态、数据加载与表单处理逻辑，确保数据流清晰可靠、易于维护。

## 1. 状态放置位置与粒度

- **状态就近管理**：
  - 优先将状态放在使用它的最近组件，而不是一开始就提升到全局。
  - 仅当多个远距离组件需要共享状态时，才考虑使用 Context 或状态库。
- **避免重复与冗余状态**：
  - 能通过现有状态或参数推导出的值不应单独存储（例如从列表中查找选中项，而不是再维护一份冗余拷贝）。
  - 避免在多个地方维护同一份业务数据的多份副本。

## 2. 异步数据加载

- **统一的数据访问层**：
  - 所有 HTTP 请求应通过统一的 API 封装模块发起，避免在组件中直接 new axios/fetch。
  - 请求 URL、方法与参数构造集中管理，组件只关心业务参数与结果。
- **完整的状态周期**：
  - 对每个异步请求，至少考虑 `loading` / `success` / `error` / `empty` 四种状态。
  - 向用户展示合适的加载中提示、错误信息与空状态，而不是静默失败或空白页面。
- **重复请求与依赖变化**：
  - 当依赖发生变化时（如选择的知识库/标签），需要重新加载数据；确保依赖列表完整且逻辑清晰。
  - 避免在同一页面中多个组件分别加载相同数据，可考虑提升到父组件或封装为共享 Hook。

## 3. 表单与用户输入

- **显式受控组件**：
  - 输入框、选择框等表单控件应作为受控组件管理，状态来源清晰。
  - 避免在 DOM 中直接读取/写入值（如通过 `document.querySelector` 操作表单）。
- **校验与错误提示**：
  - 用户操作前需满足的前置条件（如「必须选择标签/文档」）应在前端做基本校验，并给出友好提示。
  - 对于错误消息，优先使用统一的提示组件，而不是随处散落的 `alert`。
- **默认值与重置**：
  - 为下拉框/复选框等提供清晰的默认项，避免「首次加载看起来没有任何状态」。
  - 操作完成后（如提交成功）适当重置相关状态或给出下一步引导。

## 4. 状态变更与不可变性

- **保持不可变更新**：
  - 使用不可变方式更新数组和对象（`map`/`filter`/扩展运算符等），避免直接修改引用后再 set。
  - 对复杂状态可考虑使用专门的更新函数或 `useReducer` 来管理。
- **批量更新与原子性**：
  - 对多个相关状态的变更，确保更新顺序合理，避免中间状态不一致。
  - 在可能的情况下，将多个相关字段收敛为一个状态对象或结构体。

## 5. 全局状态与上下文

- **Context 的使用边界**：
  - Context 适用于少量、跨多层级的配置或 UI 状态（如侧边栏折叠状态、主题、当前用户等）。
  - 不要在 Context 中堆积大块业务数据或频繁变化的列表，以免引发大范围重渲染。
- **稳定的 Context 值**：
  - 使用 `useMemo` 确保 Context `value` 对象在依赖未变化时保持引用稳定，避免子组件无谓重渲染。

## 6. 错误处理与日志

- **对用户友好的错误**：
  - 面向用户的错误提示应可读且与操作相关，而非简单展示原始异常信息。
  - 仅在调试或开发环境下输出详细的 `console.error` 日志。
- **可恢复性**：
  - 为常见错误场景提供简单的恢复路径（重试按钮、刷新当前列表、跳转到配置页面等）。
  - 在严重错误时，可以考虑展示「安全降级」界面，而非直接白屏。
