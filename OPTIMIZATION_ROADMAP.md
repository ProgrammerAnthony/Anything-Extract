## 1. 项目现状评估

### 1.1 已完成能力盘点

AnythingExtract 经过前两个阶段（Stage 1 + Stage 2）的建设，已具备一套完整的文档结构化提取能力闭环。具体已实现的能力如下：

**文档摄入层（完整）**

系统支持 9 种文件格式（PDF、DOCX、TXT、Markdown、CSV、JSON、XLSX、PPTX、EML）的解析，每种格式配备专属解析器，并通过异步队列 Worker 将文件上传与解析处理解耦。上传后立即响应用户，解析在后台独立进程中完成，前端通过轮询获取处理状态。整个摄入流程覆盖：文件接收 → 格式验证 → 文本提取 → 递归字符分块 → Embedding 向量化 → LanceDB 存储，形成完整的 RAG 数据准备管线。

**检索与提取层（完整）**

检索层基于 Ollama Embedding 模型 + LanceDB 向量数据库，支持余弦相似度检索。在此基础上，系统叠加了 RAG 标签问题增强层：用户可选择为每个标签生成 3 个扩展检索问题，多个问题独立检索后对 chunk 去重合并，扩大召回范围。提取层支持单标签、多标签、批量三种模式，调用 Ollama LLM，要求返回结构化 JSON，内置最多 3 次重试和正则兜底解析。

**标签与知识库管理（完整）**

支持三种标签类型（单选、多选、填空），标签配置存储在 SQLite，知识库作为文档的组织容器，支持 CRUD 操作。

**Provider 抽象层（框架完整）**

LLM、Embedding、VectorDB 三类 Provider 均有抽象基类，当前各有一个 Ollama / LanceDB 实现，框架为后续接入其他 Provider 做好了准备。

**前端（功能完整，体验有待提升）**

Next.js 双侧边栏布局，包含标签管理、知识库管理、文档管理、信息提取四个主要模块。知识库详情页支持乐观插入和失败重试。信息提取页支持 RAG 增强问题的展示和刷新。

### 1.2 核心短板识别

通过全面评估，当前项目存在以下几个核心短板，直接影响用户使用体验和系统成熟度：

**短板一：用户上手门槛高**  
新用户必须手动创建每一个标签（名称、类型、描述、可选项），才能开始第一次提取。对于初次使用者，这一过程既陌生又耗时，缺少任何引导和示例，导致"看到这个产品很有用，但不知道怎么用起来"的困境。

**短板二：批量提取能力不足**  
当前批量提取接口（`/api/extract/batch`）只支持"单标签 × 多文档"，而真实企业场景（如批量审查合同、筛选大量简历）通常需要"多标签 × 多文档"的联合提取，并将结果汇聚为一张表格。这是目前功能完整性上最大的缺口。

**短板三：提取结果缺乏沉淀**  
提取结果只在当次会话中展示，用户刷新页面后结果消失。虽然 `extraction_results` 表在数据库设计中存在，但前端没有与之对接的历史查看入口，导致每次提取都是"用完即抛"，无法积累和比较。

**短板四：检索质量停留在基础阶段**  
当前检索仅有余弦向量检索（basic），Stage 4 计划引入的 Rerank 服务至今未实现，`rerank` 参数虽然在 API 中预留，但实际传入 `true` 不会触发任何额外逻辑。

**短板五：知识问答模块是空壳**  
前端 `/qa` 页面和侧边栏入口已存在，但页面内容为空。这是产品对外展示最直观的功能之一，空页面会给用户留下"未完成"的印象。

**短板六：部署门槛较高**  
项目需要用户手动配置 Python 虚拟环境、Node.js、Ollama 三个独立运行时，缺少 Docker 方案，对非技术背景的用户和希望快速试用的开发者不友好。

### 1.3 优化方向确立

基于上述评估，优化工作从两个维度同时展开：

- **用户视角（易用性）**：降低新用户上手门槛、提升批量处理能力、让结果可沉淀可对比、部署更简单
- **系统视角（技术深度）**：提升检索质量、完善数据管道、增强向量索引性能、建立质量评估体系

这两个维度不是对立关系，而是相互支撑：系统底层更强健，用户才能感受到更稳定高效的使用体验。

## 3. 优化一：提取方案模板系统

### 3.1 背景与问题

当用户第一次打开 AnythingExtract 时，面对的是一个空白的标签列表。在开始任何提取操作之前，用户需要手动完成以下步骤：进入标签管理页面、点击"新建标签"、填写标签名称、选择标签类型（单选/多选/填空）、填写标签描述（这直接影响 LLM 理解）、若是选择题则填写所有选项、重复上述操作直到所有标签创建完毕、再回到提取页面选择这些标签。

对于熟悉 RAG 提取原理的技术用户，这个过程可以接受。但对于目标用户群体中更大比例的普通用户（法务人员、HR、采购员等），这个流程既陌生又繁琐，很可能在第一步就放弃。

更深层的问题是，对于同一类文档（如合同、简历、发票），不同用户往往需要几乎相同的标签配置。这种重复劳动是没有必要的，也是产品粘性的隐性障碍：每次换一台电脑或重置数据库，所有配置都要重头开始。

### 3.2 优化目标

- 新用户在不创建任何标签的情况下，选择内置模板后 30 秒内完成首次提取
- 支持用户将自己配置好的标签组合另存为方案，下次一键复用
- 方案（TagSet）作为独立实体，与标签管理解耦，支持独立的 CRUD

### 3.3 技术方向

**3.3.1 内置场景模板库**

在后端服务层新增模板服务（`template_service.py`），内置若干常见行业场景的标签配置组合。模板数据以 Python 字典或 JSON 文件形式存储在代码仓库中，不依赖数据库，确保即装即用。

建议内置的模板场景包括但不限于：
- **合同信息提取**：合同类型（单选）、合同金额（填空）、签署日期（填空）、甲方名称（填空）、乙方名称（填空）、合同有效期（填空）、违约条款摘要（填空）
- **简历信息提取**：姓名（填空）、教育背景（填空）、工作年限（填空）、求职意向（单选）、核心技能（多选）、期望薪资（填空）
- **发票信息提取**：发票类型（单选）、开票日期（填空）、金额（填空）、税率（填空）、购买方名称（填空）、销售方名称（填空）
- **学术论文解析**：研究领域（单选）、研究方法（多选）、主要结论（填空）、数据集（填空）、模型名称（填空）
- **法律文书摘要**：文书类型（单选）、案由（填空）、当事人（填空）、判决结果（填空）、关键日期（填空）

**3.3.2 提取方案（TagSet）实体**

在数据库层新增 `tag_sets` 表，用于持久化存储用户创建和系统内置的提取方案。

一个 TagSet 包含：方案名称、描述信息、关联的标签 ID 列表（有序，保存提取时的标签排列顺序）、是否为内置模板的标记、创建和更新时间。

TagSet 与 TagConfig 是多对多的松耦合关系：同一个标签可以属于多个方案，删除方案不影响已有标签，删除标签后方案中对应条目自动失效。

**3.3.3 模板应用接口**

新增以下 API 端点：
- `GET /api/templates`：列出所有可用模板（内置 + 用户自定义），包含每个模板的简要描述和包含的标签预览
- `POST /api/templates/{template_name}/apply`：一键将模板中的标签批量创建到系统中，返回创建的标签 ID 列表（若同名标签已存在则复用，不重复创建）
- `POST /api/tag-sets`：将当前选中的标签 ID 列表保存为一个新方案
- `GET /api/tag-sets`：获取所有用户方案列表
- `DELETE /api/tag-sets/{id}`：删除方案（不删除标签本身）

**3.3.4 前端改造方向**

在信息提取页面顶部增加"选择方案"下拉选择区域，提供两种快速进入路径：
- **选择内置模板**：弹出模板卡片展示，每个卡片显示场景名称、适用文档类型、包含的标签列表预览，点击"应用"后自动将模板标签批量创建并在提取页勾选
- **选择已保存方案**：列出用户历史保存的方案，点击后一键勾选对应标签

在提取完成后，增加"保存为方案"按钮，允许用户将当前标签组合命名保存，下次复用。

### 3.4 实施路径

1. 后端定义 TagSet 数据模型，执行数据库迁移
2. 编写内置模板数据（JSON 格式），纳入代码仓库版本管理
3. 实现模板服务（`template_service.py`）和相关 API
4. 前端实现模板选择 UI 和方案管理入口
5. 编写模板文档，帮助用户了解如何扩展自定义模板

### 3.5 预期效果

新用户首次体验路径压缩为：进入应用 → 选择"合同信息提取"模板 → 上传一份 PDF → 点击提取 → 查看结构化结果。整个过程不需要理解标签系统的设计，也不需要手动配置任何内容。这是提升开源项目用户留存率最直接有效的一步。

## 4. 优化二：多标签×多文档批量提取与结果导出

### 4.1 背景与问题

在真实的企业使用场景中，用户的典型需求不是"从一份文档中提取一个字段"，而是"从 100 份合同中提取 15 个字段，汇总成一张 Excel 表格"。

当前的批量提取接口（`POST /api/extract/batch`）仅支持"单标签 × 多文档"，这意味着提取 15 个字段需要调用接口 15 次，手动拼接结果。这在工程上是低效的，在用户体验上几乎无法接受。此外，当前批量提取是同步阻塞的，100 份文档的提取会长时间阻塞 HTTP 连接，容易触发超时，也无法获知中间进度。

### 4.2 优化目标

- 支持用户选择多个标签 + 多个文档，一次提交批量提取任务
- 任务异步执行，前端可实时查看进度（完成了第几份 / 共几份）
- 任务完成后支持一键下载结果，格式覆盖 CSV、Excel、JSON
- 失败的文档在任务完成后单独列出，可重试

### 4.3 技术方向

**4.3.1 批量任务状态机**

新增独立的批量提取任务表（`batch_extract_jobs`），作为异步任务的状态载体。任务状态包括：等待中（pending）、运行中（running）、已完成（completed）、失败（failed）、部分完成（partial）。

任务创建后立即返回 `job_id` 给前端，前端通过轮询或 SSE 订阅获取进度更新。任务执行过程中，每完成一个文档的提取就更新计数器，方便前端显示"已完成 42/100"的进度。

任务元数据存储以下信息：关联的标签方案 ID、文档 ID 列表、任务创建时间、启动时间、完成时间、总文档数、已完成数、失败数、结果文件路径。

**4.3.2 并发控制策略**

批量提取的核心性能问题是并发控制。每份文档的提取包含多次 Embedding 调用（向量检索）和一次 LLM 调用，如果 100 份文档同时并发，会使 Ollama 服务超载。

推荐采用有限并发策略：允许同时处理 N 份文档（N 默认为 3-5，可配置），其余文档在内部队列中等待。这样既能充分利用系统资源，又不会造成过载。在同一文档内部，多标签的 Embedding 检索可以并行执行，LLM 调用则串行（因为多标签合并提取已经是一次 LLM 调用）。

**4.3.3 结果存储与结构**

批量提取的结果以文件形式持久化存储，不全部放在数据库中（避免大批量数据膨胀 SQLite）。文件格式为 JSONL（每行一个文档的完整提取结果），包含文档 ID、文件名、每个标签的提取值、推理过程、置信度、失败原因（若失败）等。

**4.3.4 导出格式设计**

提供三种导出格式：

- **CSV 格式**：最简洁，行为文档，列为标签字段，可直接用 Excel 打开。适合快速查看汇总数据。
- **Excel 格式**：利用项目已有的 `openpyxl` 依赖，生成多 Sheet 工作簿：第一个 Sheet 为汇总结果表，第二个 Sheet 为失败列表（包含文档名和失败原因），第三个 Sheet 为提取元数据（模型、参数、时间等）。
- **JSON 格式**：保留完整的提取过程信息，包括每个标签的 `reasoning`、`original_content`、检索来源 chunk 等调试信息。适合需要溯查提取过程的场景。

**4.3.5 前端交互设计**

在信息提取页面增加"批量提取"模式入口。进入批量模式后，用户选择标签方案（或手动勾选标签）+ 选择知识库（或多选文档）+ 确认参数（top_k、是否开启 RAG 增强），点击"开始批量提取"。

任务提交后前端跳转到批量任务详情页，实时展示进度条和当前处理的文档名。完成后显示成功/失败统计，以及"下载结果"按钮（带格式选择）。任务列表页展示所有历史批量任务，支持重新下载结果。

### 4.4 实施路径

1. 设计并创建批量任务数据库表
2. 实现批量提取核心逻辑，含并发控制和进度更新
3. 实现结果文件生成（CSV / Excel / JSON 三种格式）
4. 实现任务状态查询 API 和结果下载 API
5. 前端实现批量模式 UI、任务进度展示、结果下载

### 4.5 预期效果

处理 100 份合同、15 个标签的场景：用户提交任务后，系统在后台全自动处理（约 15-30 分钟，取决于硬件），期间用户可关闭页面，完成后重新进入即可下载结果 Excel。这将 AnythingExtract 从"演示工具"升级为真正可以投入企业工作流的生产工具。

## 5. 优化三：提取结果持久化与历史版本管理

### 5.1 背景与问题

提取结果是这个产品最核心的产出物，但当前的处理方式令人遗憾：用户完成一次提取后，如果刷新页面，结果就消失了。虽然数据库中已有 `extraction_results` 表，但前端没有与之对接的历史查看入口。

这导致了几个实际问题：用户调整参数后重新提取，无法和上一次结果对比；用户想展示某次提取结果，必须保持浏览器窗口打开；团队协作时无法分享特定提取结果；无法积累人工审核的反馈数据用于后续优化。

### 5.2 优化目标

- 所有提取结果自动持久化，用户可随时查看历史记录
- 支持同一文档、同一标签的不同参数提取结果进行横向对比
- 支持用户对提取结果进行人工校验（确认正确 / 修改为正确值）
- 人工校验数据形成反馈闭环，可用于未来的提取质量分析

### 5.3 技术方向

**5.3.1 提取结果模型增强**

在现有 `extraction_results` 表基础上补充以下字段：
- 提取参数快照（JSON 字符串）：记录此次提取使用的 top_k、是否开启 RAG 增强、使用的策略等参数
- 置信度分数（浮点数，0-1）：来自 LLM 自评或后续计算（详见优化十二）
- 用户校验状态（枚举）：未校验、已确认正确、已修改
- 用户修正值（文本）：用户认为正确的值
- 校验时间戳

为每条提取结果生成一个可访问的唯一链接（基于 UUID，`/extractions/{id}`），方便分享和引用。

**5.3.2 文档提取历史视图**

在文档详情页（`/documents/[id]`）新增"提取历史"Tab，展示该文档所有历史提取记录。

每条历史记录以卡片形式展示：提取时间、使用的方案（标签组合）、参数摘要（top_k 等）、各标签结果概览（只展示值，不展示详情）、用户校验状态徽标。

点击单条历史记录进入详情，可查看完整的提取结果、检索来源、推理过程等信息。

**5.3.3 结果对比视图**

在文档提取历史页面，支持多选两条历史记录进行并排对比。对比视图采用左右分栏布局：左栏为记录 A 的参数和结果，右栏为记录 B 的参数和结果。对于相同标签，若两次提取结果不同，则高亮标注差异，方便用户评估不同参数配置的效果差异。

这个功能对于调优提取效果特别有价值：用户可以对比"top_k=3 vs top_k=8"、"开启 RAG 增强 vs 不开启"等不同配置的实际效果。

**5.3.4 人工校验工作流**

在提取结果展示中，为每个标签的提取值增加操作按钮：
- "确认"按钮：标记当前值为正确，记录校验状态
- "修改"按钮：弹出编辑框，允许用户输入正确值后保存

校验数据持久化到数据库，可通过管理 API 导出，用于分析模型的提取准确率，也为未来的 few-shot 示例优化提供高质量标注数据。

**5.3.5 全局提取历史页面**

新增全局提取历史页面（`/extractions`），以表格形式展示所有历史提取记录，支持按文档名、日期范围、校验状态过滤，支持批量导出。

### 5.4 实施路径

1. 扩展 `extraction_results` 表结构，执行数据库迁移
2. 确保每次提取都完整持久化结果（当前可能只在批量模式下保存）
3. 实现历史查询 API（按文档 ID、按时间范围、支持分页）
4. 前端实现文档详情页的"提取历史"Tab
5. 实现结果对比视图
6. 实现人工校验交互和后端校验数据写入

### 5.5 预期效果

提取结果从"用完即抛"变为"可积累的数据资产"。用户可以随时回溯历史结果，进行参数对比实验，并通过人工校验逐步建立起一份针对自身业务场景的高质量标注数据集，为后续的 Prompt 优化和模型评估提供基础。

## 6. 优化四：智能分块策略升级

### 6.1 背景与问题

文本分块（Chunking）是 RAG 系统中最被低估、影响却最大的环节之一。当前系统使用的 `RecursiveCharacterTextSplitter` 是一种通用的、基于字符数的分块方法，对普通文本段落效果尚可，但对于结构化内容存在明显缺陷：

- **表格被切断**：一张包含 20 行数据的 Excel 表格，按字符长度分块后可能被切成多段，每段都缺少表头，导致 LLM 无法理解列含义
- **标题层级丢失**：Markdown 或结构化报告中，标题和其下方的正文段落可能分属不同 chunk，检索时只能找到正文，失去了"这是第三章第二节的内容"这一重要语境
- **语义被割裂**：一段论述同一主题的文字，可能因为字数恰好超过 chunk_size 而被切成"语义前半段"和"语义后半段"，分开看都不完整

这些问题直接导致：对于含表格的合同或财务报告，关键字段所在的表格行被切碎，检索时虽然命中了附近的 chunk，但 LLM 收到的内容是残缺的，提取质量大幅下降。

### 6.2 优化目标

- 针对不同文档类型，使用最适合的分块策略，而非统一的字符分块
- 保护表格结构的完整性，表格作为整体进入单个 chunk
- 建立父子 chunk 机制，检索时用小 chunk 精准命中，返回时用大 chunk 提供完整语境
- 自动过滤低质量碎片 chunk，减少噪声对 LLM 的干扰

### 6.3 技术方向

**6.3.1 文档类型感知的分块路由**

在文本分块器（`text_splitter.py`）中引入路由机制，根据文档类型选择对应的分块策略：

- **Markdown 和结构化报告**：按标题层级（H1 → H2 → H3）递归分块，每个 chunk 尽量对应一个完整的标题节（标题 + 其下所有段落），当节内内容过长时再按段落切分
- **Excel 和 CSV 数据表**：以表格为单位分块，每个 chunk 包含表头行 + 若干数据行，确保每个 chunk 都是可以独立理解的完整表格片段
- **含表格的 PDF**：PDF 解析时（pdfplumber 已有表格检测能力）将表格区域和文本区域分开处理，表格作为独立 chunk，文本按段落分块
- **普通段落文本**：保持现有的 `RecursiveCharacterTextSplitter` 逻辑，这对连续叙述性文本效果良好

**6.3.2 父子 Chunk 架构**

参考 LangChain 的 `ParentDocumentRetriever` 设计，建立两个粒度级别的 chunk：

- **小 chunk（检索单元）**：较短，约 200-300 字符，用于向量化和检索。小 chunk 语义聚焦，向量表示精确，检索命中率高。
- **大 chunk（提取单元）**：较长，约 800-1200 字符，涵盖小 chunk 所在的完整语义段落。LLM 进行信息提取时，使用大 chunk 作为上下文，信息更完整。

两者通过 `parent_chunk_id` 字段建立关联。向量化和检索只用小 chunk，检索命中后系统自动返回对应的大 chunk 内容给 LLM。这一机制可以同时提升检索精度和提取完整性。

在 LanceDB 元数据中新增 `parent_chunk_id` 字段，检索结果后处理时，根据 `parent_chunk_id` 去重（多个小 chunk 命中同一 parent 时，parent 只返回一次），并取回大 chunk 的完整内容。

**6.3.3 Chunk 质量过滤**

分块完成后增加质量过滤步骤，自动丢弃以下类型的低质量 chunk：
- 内容长度小于 20 个字符的碎片（通常是页码、标注、图片说明等）
- 纯数字或纯标点符号的内容（目录页码行、装饰性分割线等）
- 信息密度极低的重复内容（如多行重复的表单空白行）

这一步骤可减少 10%-30% 的无效向量，节省向量化时间，并降低 LLM 收到噪声内容的概率。

**6.3.4 配置化支持**

分块策略应作为可配置参数暴露，允许高级用户根据自己的文档特点调整：
- 小/大 chunk 的目标大小（字符数）
- 是否开启父子 chunk 模式（默认开启）
- 表格行合并策略（每个 chunk 包含多少行数据）
- 质量过滤的最低长度阈值

### 6.4 实施路径

1. 扩展 `document_parser.py` 的表格提取能力，返回结构化的表格对象
2. 在 `text_splitter.py` 中实现路由逻辑和各类型分块策略
3. 在 LanceDB 向量元数据中增加 `parent_chunk_id` 字段
4. 修改 `retrieval_service.py` 的结果后处理，命中小 chunk 后自动返回 parent 内容
5. 为历史文档提供重新分块和向量化的"重建"功能（管理员操作）

### 6.5 预期效果

含表格的合同和财务报告提取准确率预计提升 20%-40%。标题层级感知分块让检索结果更具语境，减少"答案在文档里但 LLM 给不出正确结果"的情况。父子 chunk 机制在不降低检索精度的前提下，为 LLM 提供更完整的上下文，特别是对填空型标签的效果改善明显。

## 7. 优化五：Rerank 精排服务落地（完成 Stage 4）

### 7.1 背景与问题

当前检索链路：查询向量化 → 余弦相似度向量检索 → 返回 top_k 结果 → 直接喂给 LLM。

这条链路存在一个系统性问题：向量相似度是一种粗粒度的语义匹配，它衡量的是"查询向量和文档向量在高维空间的距离"，而不是"这个文档片段真的能回答这个问题"。两者是相关的，但不等同。

实践中经常出现：向量相似度最高的几个 chunk，并不包含真正需要提取的信息，而真正包含信息的 chunk 在排名第 6-10 位。如果 top_k=5，那么正确答案所在的 chunk 就被截断了。

Stage 4 计划引入 Rerank 正是为了解决这个问题。API 已预留 `rerank: bool` 参数，但实际逻辑尚未实现。本项目已规划该阶段，现在是落地的时机。

### 7.2 优化目标

- 实现可工作的 Rerank 机制，当用户开启时确实提升检索质量
- 提供至少一种本地可运行（无需额外服务）的 Rerank 方案
- Rerank 作为可选功能，不影响不开启时的现有流程
- 与现有 Provider 抽象体系保持一致，便于未来替换不同的 Rerank 实现

### 7.3 技术方向

**7.3.1 Rerank 的本质**

Rerank（重排序）是在初步召回之后，对召回结果进行二次精细排序的过程。与向量检索的"用向量距离代理相关性"不同，Rerank 模型直接对"查询 - 文档"对进行相关性打分，精度更高，但计算开销也更大（因此不能替代向量召回，只能在召回之后作为精排步骤）。

标准的混合检索 + Rerank 流水线如下：初步召回（向量检索，获取 top_k×2 个候选） → Rerank 精排（对所有候选打分并重新排序） → 取重排后的前 top_k 个 → 喂给 LLM。

**7.3.2 BM25 稀疏检索精排（推荐首选）**

BM25 是经典的词频-逆文档频率稀疏检索算法，与向量检索互补：向量检索擅长捕捉语义相似性，BM25 擅长捕捉关键词精确匹配。

实现方案：在文档摄入时，同步构建 BM25 索引（存储在内存或本地文件）。检索时，向量检索和 BM25 检索分别获取 top_k×2 个结果，通过倒数排名融合（Reciprocal Rank Fusion, RRF）将两组结果合并为一个统一排名，取前 top_k。

这种方案完全本地运行，不需要额外的 GPU 资源，仅依赖轻量级的 `rank_bm25` 库（纯 Python 实现），对现有系统的资源要求几乎没有影响。

**7.3.3 Cross-Encoder 深度精排（可选高级方案）**

Cross-Encoder 是更精确的 Rerank 方案，它将查询和文档拼接后输入一个小型 BERT 模型，直接输出相关性分数。精度比 BM25 更高，但需要在本地运行一个小型 Transformer 模型。

适合资源较充裕的用户。推荐使用 `cross-encoder/ms-marco-MiniLM-L-6-v2`（约 80MB），可通过 `sentence-transformers` 库加载。

**7.3.4 Rerank Provider 抽象**

与现有 Provider 体系保持一致，新增 `providers/rerank/` 模块：定义 `RerankProvider` 基类，提供 BM25 和 Cross-Encoder 两种实现，并提供空实现（passthrough，不做重排，用于禁用时的占位）。

系统配置中新增 Rerank 相关参数：是否默认开启、使用哪种 Rerank 方法、召回候选数（初始召回 top_k 的倍数）。

**7.3.5 前端控制**

信息提取页的"高级参数"面板（当前已有 RAG 增强开关）中新增 Rerank 开关和方法选择（BM25 / Cross-Encoder / 无）。对普通用户，默认显示一个简单的"开启精排"开关；高级用户可展开查看方法选择和参数配置。

### 7.4 实施路径

1. 新增 `providers/rerank/` 模块，定义基类和 BM25 实现
2. 在文档摄入时同步构建 BM25 索引（以知识库为粒度）
3. 修改 `retrieval_service.py`，当 `rerank=True` 时执行混合检索 + 重排流水线
4. 将 Rerank 实现与 API 中已有的 `rerank` 参数连通
5. 前端在高级参数面板中暴露 Rerank 控制
6. 可选：实现 Cross-Encoder Rerank 作为进阶选项

### 7.5 预期效果

对于标签描述中包含明确关键词的场景，开启 Rerank 后命中正确 chunk 的概率显著提升，预计减少 20%-30% 的"向量相似但内容无关"误召回。用户感受到的变化是：提取结果中"原文证据"与标签描述更加匹配，LLM 给出的 reasoning 更有说服力。

## 8. 优化六：知识问答模块完整实现

### 8.1 背景与问题

当前系统的"提取"模式要求用户预先配置好标签（字段定义），再对文档进行结构化抽取。这种模式非常适合"我知道要找什么"的场景（如合同审查、发票录入）。

但还有大量场景是"我不知道要找什么，我想探索性地了解文档内容"，例如："这份技术报告的主要结论是什么？"、"这批合同中有没有超过 5 年的长期合同？"、"这篇论文的实验设计和我们公司的需求有什么关联？"

这类场景不适合标签提取模式，需要自然语言问答（QA）模式。前端已有 QA 页面的路由（`/qa`）和侧边栏入口，但页面内容为空。实现知识问答模块是填补这一空白的关键。

### 8.2 优化目标

- 用户可以用自然语言向指定知识库提问，系统返回基于知识库内容的回答
- 支持多轮对话（系统记住对话上下文，用户可追问）
- 每条回答附带引用来源（引用了哪个文档的哪个段落），支持溯源
- 回答以流式方式返回，避免长时间等待白屏

### 8.3 技术方向

**8.3.1 单轮问答核心逻辑**

单轮问答的核心流程与现有提取逻辑高度相似，差别在于"查询"变成了用户的自然语言问题，而不是预定义的标签描述。

流程：接收用户问题 → 向量化问题 → 在指定知识库中检索 top_k 个相关 chunk → 构建问答 Prompt（系统提示 + 检索到的上下文 + 用户问题）→ 调用 LLM → 流式返回回答。

这个流程可以复用现有的 `retrieval_service.py` 和 `llm_processing_service.py`，核心业务逻辑量不大。

**8.3.2 多轮对话上下文管理**

多轮对话的挑战在于上下文窗口管理：LLM 有 token 限制，历史对话不能无限累积。

推荐策略：保留最近 N 轮（默认 5 轮）的完整对话记录，当超过限制时，使用 LLM 对早期历史进行摘要压缩，保留摘要替代原始内容。每次发送给 LLM 的上下文格式为：历史摘要（若有）+ 最近 N 轮完整对话 + 当前检索到的文档片段 + 用户当前问题。

**8.3.3 对话持久化**

新增两张数据库表：
- `chat_sessions`：对话会话，关联到知识库，包含会话标题（可自动从首个问题生成）、创建时间
- `chat_messages`：对话消息，每条消息包含角色（用户/助手）、内容、引用的来源 chunk（JSON 数组）、创建时间

对话历史持久化后，用户可以随时返回查看之前的问答记录。

**8.3.4 引用溯源展示**

每条 LLM 回答附带"引用来源"区域，展示本次回答参考了哪几个文档片段，每个片段显示：来源文档名称、页码、内容摘要（前 100 字符）。用户点击来源可跳转到对应文档详情页（未来接入文档预览后可直接定位到页面）。

**8.3.5 前端 QA 页面设计**

标准的聊天气泡界面，分为三个区域：
- 左侧：知识库选择面板 + 历史会话列表，支持新建会话和切换会话
- 中间：对话内容区，用户消息和 AI 回答交替显示，AI 回答支持 Markdown 渲染（代码块、列表、标题等格式化展示）
- 右侧（可折叠）：当前消息的引用来源详情

回答过程中，中间区域显示流式打字效果，用户感受到实时响应，不会面对长时间的白屏等待。

### 8.4 实施路径

1. 新增对话会话和消息数据模型
2. 实现单轮问答 API（含流式版本）
3. 实现多轮对话上下文管理逻辑
4. 实现历史会话查询 API
5. 前端实现完整的 QA 聊天界面
6. 对接流式 API，实现打字机效果

### 8.5 预期效果

QA 模块实现后，AnythingExtract 具备了"结构化提取"和"自由问答"两种互补的知识获取模式。前者适合有明确字段定义的批量处理，后者适合探索性分析和即时查询。两种模式共享同一套知识库和文档资产，用户无需做任何重复配置。

## 9. 优化七：多 Provider 支持与云端 API 接入

### 9.1 背景与问题

当前 LLM 和 Embedding 都只有 Ollama 实现。Ollama 是本地化部署的理想选择，但存在几个限制：需要用户本地有足够算力（特别是 GPU），初次拉取模型需要等待，小模型（如 3B 参数）在复杂提取任务上效果不如 GPT-4o 等大模型。

目前大量用户已经有了 OpenAI API Key、DeepSeek API Key 或通义千问等国内云端 API 的访问权限。对这些用户，直接使用云端 API 是比本地 Ollama 更便捷的选择，且提取质量通常更高。

Provider 抽象层已经设计好了基类，现在需要实现具体的替代方案，真正发挥这套架构的价值。

### 9.2 优化目标

- 通过配置文件切换 LLM Provider，不需要修改代码
- 支持 OpenAI 官方 API 和任何 OpenAI 兼容格式的 API（涵盖 DeepSeek、通义、GLM 等主流国内模型）
- 前端提供 Provider 配置页面，用户可在 UI 中切换和测试连接
- API Key 等敏感信息安全存储，不在日志中暴露

### 9.3 技术方向

**9.3.1 OpenAI 兼容 LLM Provider**

大多数主流 LLM API 服务（包括 OpenAI、DeepSeek、通义千问、智谱 GLM、月之暗面 Kimi 等）都提供了与 OpenAI API 格式兼容的接口，只需修改 `base_url` 和 `api_key` 即可切换。

只需实现一个 `OpenAICompatibleProvider` 类，通过不同的 `base_url` 参数覆盖所有这些服务：
- OpenAI 官方：`https://api.openai.com/v1`
- DeepSeek：`https://api.deepseek.com`
- 通义千问：`https://dashscope.aliyuncs.com/compatible-mode/v1`
- 智谱 GLM：`https://open.bigmodel.cn/api/paas/v4`
- Ollama（OpenAI 兼容模式）：`http://localhost:11434/v1`

实现时使用 `openai` Python SDK 的异步客户端，支持流式和非流式两种调用方式。

**9.3.2 OpenAI 兼容 Embedding Provider**

类似地，OpenAI 的 Embedding API 也被广泛兼容。实现 `OpenAICompatibleEmbeddingProvider`，通过 `base_url` 切换到不同的 Embedding 服务。

需要注意的是，不同 Embedding 模型输出的向量维度不同（如 `text-embedding-3-small` 输出 1536 维，`nomic-embed-text` 输出 768 维）。系统已有 `model_dimensions.py` 维度映射表，切换 Embedding 模型时需要检测维度变化，必要时重建向量索引。

**9.3.3 Provider 工厂与热切换**

在后端核心层新增 Provider 工厂模块（`core/provider_factory.py`），负责根据配置实例化对应的 Provider。所有需要使用 LLM 或 Embedding 的服务，通过工厂获取 Provider 实例，而不是直接 import 具体实现类。

在运行时，当用户通过系统配置页面更新 Provider 配置后，工厂重新实例化 Provider（无需重启服务）。

**9.3.4 前端配置界面**

系统配置页（`/system`）扩展以下内容：

LLM 配置区域：Provider 类型下拉（Ollama / OpenAI 兼容），显示对应的参数输入项（Ollama 只需 base_url 和模型名，OpenAI 兼容额外需要 api_key），以及"测试连接"按钮（后端验证连通性和 API Key 有效性后返回结果）。

Embedding 配置区域：与 LLM 配置类似，增加"切换模型需要重建向量索引"的提示。

**9.3.5 API Key 安全存储**

API Key 不应明文存储在 `.env` 文件中（可能被误提交到 Git）。生产使用建议：
- 优先从环境变量读取
- 支持从加密的本地密钥文件读取（使用系统密钥环或简单的 AES 加密）
- 在日志中对 API Key 做脱敏处理（只显示前 4 位和后 4 位）

### 9.4 实施路径

1. 实现 `OpenAICompatibleProvider`（LLM 和 Embedding 各一个）
2. 实现 `ProviderFactory`，实现基于配置的动态实例化
3. 扩展系统配置 API，支持 Provider 配置的读写和连通性测试
4. 前端实现 Provider 配置 UI
5. 文档中补充各主流 API 的配置示例

### 9.5 预期效果

项目对不同技术背景用户的覆盖范围大幅扩大：本地算力充裕的用户继续使用 Ollama；希望更高质量提取的用户可以接入 GPT-4o 或 DeepSeek；国内用户可以方便地使用通义千问等国内大模型。开源项目的受众从"有本地 GPU 的技术人员"扩展到"有任何大模型 API 访问权限的用户"。

## 10. 优化八：文档预览与提取结果溯源定位

### 10.1 背景与问题

提取结果的"来源"（`sources`）字段当前只展示 chunk 的文字内容和相似度分数，用户看到的是："这个结果是从文档的某个段落提取出来的"，但无法直观地看到这段内容在原文档中的具体位置。

对于需要人工审核的场景（如合同风险审查、财务数据核实），用户通常需要核对原文——他们必须打开原始文档，搜索关键词，找到对应位置，这一过程打断了工作流，效率低下。

更深的问题是：当提取结果存疑时（比如 LLM 给出了一个置信度较低的答案），用户需要亲自检查"引用的原文"是否真的支持这个答案。如果应用内无法查看原文，用户的验证路径就断了。

### 10.2 优化目标

- 在应用内展示 PDF 文档内容，不需要用户打开外部程序
- 提取结果的每条来源可以直接定位到原文档的对应页面
- 在文档预览中高亮显示与提取结果相关的文本区域
- 用户可以在预览中直接验证"AI 说的和原文一致吗"

### 10.3 技术方向

**10.3.1 PDF 在线预览**

在文档详情页（`/documents/[id]`）集成 PDF 渲染能力。推荐使用 `react-pdf`（基于 `pdfjs-dist`），这是 Next.js 生态中最成熟的 PDF 渲染方案，完全在浏览器端运行，无需后端额外处理。

前端展示效果：PDF 按页渲染为 Canvas，用户可翻页、缩放。页面加载时先渲染当前页，其他页懒加载，避免大文件初始化过慢。

**10.3.2 位置元数据记录**

实现这一功能的关键是在文档解析时记录每个 chunk 在原文档中的位置信息。

对于 PDF 文档：`pdfplumber` 在提取文本时可以同时获取文字的坐标信息（x0, y0, x1, y1），记录每个 chunk 起始和结束字符在页面上的边界框（bounding box）。这个信息在向量元数据中记录为 `bbox` 字段。

对于其他格式（DOCX、TXT 等）：记录 chunk 在文档全文中的字符偏移量（`char_offset_start`, `char_offset_end`），用于文本搜索和高亮。

这些位置信息在文档解析阶段（`document_parser.py`）采集，存入向量元数据，在检索结果中随 source chunk 一起返回。

**10.3.3 来源跳转与高亮**

在提取结果展示区域，每条 source chunk 的来源卡片显示：文档名 + 第几页，以及"查看原文"链接按钮。点击后跳转到文档详情页，URL 携带页码参数（`/documents/[id]?page=3`），前端自动定位到对应页面并高亮相关文本区域。

高亮实现方式：前端在渲染 PDF 页面时，在 Canvas 上叠加一个透明的 SVG/HTML 层，根据 bbox 信息绘制高亮矩形框。

**10.3.4 内联预览（提取结果侧边栏）**

进阶功能：在信息提取页面右侧增加文档预览侧边栏，用户点击某条 source chunk 时，侧边栏实时展示对应的 PDF 页面并高亮相关内容。用户无需跳转页面，直接在提取界面完成"查看原文—验证结果"的闭环。

### 10.4 实施路径

1. 在 `document_parser.py` 的 PDF 解析逻辑中增加 bbox 坐标采集
2. LanceDB 元数据扩展 `bbox` 和 `char_offset` 字段
3. 检索结果 API 响应中包含位置元数据
4. 前端文档详情页集成 `react-pdf`，支持页码跳转
5. 实现 PDF 高亮层（Canvas 叠加）
6. 提取结果展示中接入"查看原文"功能

### 10.5 预期效果

对于审核场景，用户验证一条提取结果的时间从"切换到文档查找内容（约 1-3 分钟）"缩短到"点击查看原文（约 3 秒）"。这不仅是效率的提升，更是信任的建立：用户能轻松验证 AI 的每一个提取结果，对整个系统的置信度因此提高。

## 11. 优化九：向量索引与检索性能优化

### 11.1 背景与问题

当前向量检索存在两个性能隐患，在小数据量下不明显，但随着文档数量增长会逐渐显现：

**问题一：无 ANN 索引，线性扫描**  
LanceDB 当前配置没有启用近似最近邻（ANN）索引，每次检索都是对全表向量做暴力线性扫描，时间复杂度为 O(n)。当 chunk 总数超过 10 万时（约 1000 份中等长度 PDF），检索延迟会明显增加。

**问题二：所有知识库共享一张向量表**  
所有文档的向量都存储在同一个 LanceDB 表（`documents.lance`），依靠 `document_id` 字段过滤来实现"只检索某个知识库的文档"。这在数据量增大时效率低下，删除知识库时也需要逐条删除向量记录（而不是直接删除表）。

**问题三：Embedding 调用串行化**  
文档摄入时对每个 chunk 串行调用 Ollama Embedding API，一份 100 个 chunk 的文档需要 100 次顺序请求，处理时间随文档长度线性增长。

### 11.2 优化目标

- 文档规模增长到数千份时，检索延迟保持在可接受范围（< 1 秒）
- 知识库的向量数据管理更高效（按知识库隔离存储）
- 文档摄入速度提升，减少用户等待时间

### 11.3 技术方向

**11.3.1 LanceDB IVF-PQ 向量索引**

LanceDB 原生支持 `create_index()` 方法，当前代码未调用。可在以下时机自动构建索引：
- 每次摄入新文档后，当表中向量数量超过设定阈值（如 5000 个）时，异步触发索引重建
- 提供手动触发索引重建的管理 API（适合大批量上传后手动优化）

IVF（倒排文件索引）+ PQ（乘积量化）组合：IVF 将向量空间划分为若干分区，查询时只扫描最相关的几个分区；PQ 对向量做有损压缩，大幅降低内存占用。组合使用后，检索时间复杂度从 O(n) 降至近似 O(√n)，在 100 万向量规模下仍有毫秒级响应。

**11.3.2 知识库级别向量分表**

将向量存储策略从"全局单表 + document_id 过滤"改为"知识库级别独立表"。每个知识库对应一张 LanceDB 表，表名格式为 `kb_{knowledge_base_id}`。

优势：
- 知识库内检索不需要扫描其他知识库的向量，效率直接翻倍（若有 N 个知识库）
- 删除知识库时直接删除整张 LanceDB 表，操作效率从 O(文档向量数) 降为 O(1)
- 每个知识库可以独立配置索引参数（如向量维度、距离度量方式）

迁移时需要注意历史数据的兼容：提供数据迁移脚本，将现有全局表的数据按 `knowledge_base_id` 分拆到各知识库表。

**11.3.3 Embedding 批量并行化**

Ollama Embedding API 支持批量请求（一次请求多个文本）。将现有的逐 chunk 串行请求改为批量并行：将文档的所有 chunk 分批（每批 20-32 个），并行发送多个批量请求，合并结果。

这一优化可使 100 个 chunk 的文档向量化时间从约 30 秒降至约 5 秒，大幅提升用户等待体验（特别是选择"即时处理"模式时）。

**11.3.4 Chunk 级别向量缓存**

当前向量缓存以整个文档内容的 SHA256 为键，文档内容改变一个字符就使全部缓存失效。改为以单个 chunk 内容的 SHA256 为键进行缓存。

这样，当用户重新上传更新了某几页的文档时，未修改的页面对应的 chunk 可以复用缓存，只对修改的内容重新计算向量。在文档版本迭代的场景（如合同修订版）中，缓存命中率可从 0% 提升到 80%-90%，大幅减少 Embedding API 调用量。

### 11.4 实施路径

1. 实现知识库分表逻辑，修改 `lancedb.py` Provider 的表名策略
2. 提供历史数据迁移脚本
3. 实现向量索引自动构建（阈值触发 + 手动触发 API）
4. 将 Embedding 调用改为批量并行模式
5. 将向量缓存粒度从文档级改为 chunk 级

### 11.5 预期效果

在 1000 份文档（约 50 万 chunk）的规模下，检索延迟保持在 200ms 以内（当前无索引时约 2-5 秒）。文档摄入速度提升 3-5 倍。知识库管理操作（删除、清空）从"等待几十秒"变为"即时完成"。

## 12. 优化十：Docker 一键部署与开源工程化

### 12.1 背景与问题

当前安装文档（`install.sh` / `install.bat`）要求用户依次完成以下步骤：安装 Python 3.10+、创建虚拟环境、安装 Python 依赖、安装 Node.js 18+、安装 npm 依赖、安装并启动 Ollama、下载 LLM 模型、配置 `.env` 文件、启动后端和前端。

这些步骤对技术人员并不困难，但对非技术用户（如法务、HR 等最终用户群体）来说门槛过高。对于希望快速试用的技术用户，也需要花费 15-30 分钟完成环境配置，很多人会在这一步放弃。

作为一个开源项目，部署体验是影响 GitHub Star 数和贡献者数量的重要因素。"clone → 一条命令 → 跑起来"是开源项目的黄金标准。

### 12.2 优化目标

- 提供 Docker Compose 方案，用户只需 `docker compose up` 即可启动完整服务
- 无需手动配置 Python 环境和 Node.js 环境
- Ollama 在宿主机运行，容器通过网络访问（避免 GPU 资源在容器内难以配置的问题）
- 建立基础的 CI/CD 流水线，确保代码提交不破坏构建

### 12.3 技术方向

**12.3.1 Docker 镜像构建**

为后端和前端分别构建 Docker 镜像：

后端镜像基于 `python:3.11-slim`，采用多阶段构建（第一阶段安装依赖，第二阶段复制必要文件），最终镜像只包含运行所需内容，控制镜像大小。需要注意 `unstructured` 等解析库有大量依赖，镜像大小预计 1-2GB，可通过 slim 模式和精选依赖控制。

前端镜像采用 Node.js 构建阶段 + Nginx 运行阶段的多阶段构建。构建阶段执行 `npm run build` 生成静态文件，运行阶段使用轻量级 Nginx 托管，最终镜像约 30-50MB。

**12.3.2 Docker Compose 编排**

`docker-compose.yml` 包含三个核心服务：
- `backend`：FastAPI 后端，挂载 `./storage` 目录实现数据持久化，通过环境变量接收 Ollama 地址

- `frontend`：Nginx 托管的 Next.js 静态资源，配置反向代理到 backend

Ollama 在宿主机运行，容器通过 `host.docker.internal`（Docker Desktop）或 `172.17.0.1`（Linux Docker Bridge 网关）访问宿主机上的 Ollama 服务，避免复杂的 GPU 资源映射配置。

**12.3.3 环境变量向导**

提供交互式配置向导（`setup.py` 或 Shell 脚本），引导用户回答几个关键问题后自动生成 `.env` 配置文件：

向导结束时输出完整的启动命令，用户复制粘贴即可。

**12.3.4 GitHub Actions CI 流水线**

建立基础的自动化验证流水线（`.github/workflows/ci.yml`），在每次代码提交和 Pull Request 时执行：

CI 流水线的存在对开源贡献者友好：每个 PR 都有自动验证，降低维护者的审查负担，也让贡献者能即时获知代码是否符合规范。

**12.3.5 开发者文档完善**

补充以下开发者文档：

- 架构决策记录（ADR）：记录关键技术选型的原因（如为什么用 LanceDB 而不是 ChromaDB、为什么用 SQLite 而不是 PostgreSQL），帮助新贡献者理解设计背景

## 13. 优化十一：流式提取与批量任务实时进度

### 13.1 背景与问题

当前信息提取的用户体验存在"黑盒感"：用户点击提取按钮后，界面没有任何反馈，经过数秒甚至数十秒的等待，结果突然出现。在等待过程中，用户无从判断：系统是在正常处理、还是卡住了、还是出错了。

这种体验在移动互联网时代已经不可接受。流式输出（用户看到 AI 逐字打出答案）和进度条（用户知道"还剩多少"）是现代 AI 产品的标配体验元素。

`extract_stream.py` 在后端已存在基础实现，但前端几乎未对接。这个优化的重点在于将现有能力在前端完整展现出来。

### 13.2 优化目标

- 单次提取过程中，用户能实时看到每个标签的处理进度（检索中 → 检索完成 → LLM 处理中 → 结果返回）
- 批量提取任务有实时进度条和当前处理文档名展示
- 所有长时操作都有明确的进度反馈，消除"黑盒感"

### 13.3 技术方向

**13.3.1 多标签分阶段 SSE 流**

改进流式提取 API，按处理阶段实时推送事件，而不是等全部完成才返回：
- 标签处理开始事件：通知前端某标签开始处理，前端显示"检索中"状态
- 检索完成事件：携带召回的 chunk 数量，前端显示"已找到 N 个相关片段"
- LLM 处理中事件：前端显示 loading 动画
- 标签结果事件：携带具体提取结果，前端立即渲染该标签的值
- 全部完成事件：携带总耗时等统计信息

各标签的处理可以并行（Embedding 检索并行，LLM 调用串行），因此用户看到的效果是：多个标签几乎同时开始检索，检索完成后依次（或近似同时）出现结果。

**13.3.2 前端骨架屏与渐进渲染**

提取结果区域的渲染策略改变：点击提取按钮后，立即渲染所有标签的占位骨架屏（灰色条状占位符），传达"正在处理"的信息。随着 SSE 事件到达，骨架屏逐个被替换为实际结果，最终呈现为完整的提取结果表格。

对于 LLM 输出的 reasoning 文字，可以实现打字机效果（逐字符渲染），模拟 ChatGPT 的回答体验，增加产品的现代感。

**13.3.3 批量任务进度 SSE**

批量提取任务提交后，前端订阅该任务的 SSE 进度流（`GET /api/extract/batch/{job_id}/progress`）：每隔若干秒推送一次进度更新，包含已完成数量、总数量、当前处理的文档名称、最近完成的文档提取结果预览。

进度页面展示：进度条（百分比）、已完成 / 总数 / 失败数统计、最近 5 条完成记录滚动展示、预计剩余时间（基于当前速度估算）。

**13.3.4 Worker 处理进度实时推送**

文档摄入（解析 → 向量化）过程同样可以细化进度反馈：由目前的"processing"单一状态，拆分为更细粒度的状态：parsing（解析中）→ chunking（分块中）→ embedding（向量化中，x/total chunks）→ completed。

前端知识库文档列表页的轮询升级为 SSE 订阅，实时显示向量化进度百分比（"已向量化 35/100 个片段"），让用户对大文件的处理进度有准确预期。

### 13.4 实施路径

1. 扩展流式提取 API，增加中间阶段事件推送
2. 实现批量任务进度 SSE 端点
3. 前端提取结果区域改造为骨架屏 + 渐进渲染
4. 批量任务页面接入进度 SSE
5. 文档摄入状态细化和前端进度展示

### 13.5 预期效果

"黑盒感"完全消除。用户点击提取后，看到骨架屏立即出现、各标签依次完成检索、结果逐步呈现，整个过程透明可见，既增加了产品的现代感，也显著降低了用户的焦虑感（即使处理时间不变，有进度反馈的等待体验比无反馈好 5-10 倍）。

## 14. 优化十二：提取质量评估与 Prompt 调优系统

### 14.1 背景与问题

当前系统没有任何机制帮助用户了解"提取质量怎么样"或"哪种参数配置更好"。用户只能凭主观感受判断，或者通过人工抽检来评估。这对于需要精确控制提取质量的专业用户来说是一个显著短板。

更进一步，Prompt 的质量直接决定提取效果，但当前的系统提示词硬编码在服务层代码中，用户如果想针对自己的特定业务场景调整 Prompt，必须修改源代码并重启服务，这对非技术用户完全不可行。

### 14.2 优化目标

- 为每次提取结果提供置信度评估，帮助用户快速识别需要人工审核的条目
- 支持用户对比不同参数组合（top_k、RAG 增强开关等）的提取效果
- 为高级用户提供 Prompt 可视化查看和自定义修改能力
- 建立长期的提取质量跟踪机制

### 14.3 技术方向

**14.3.1 LLM 自评置信度**

在 Prompt 设计中增加置信度自评要求：要求 LLM 在返回提取结果的同时，对每个标签的提取值给出一个 0-1 之间的置信度分数，并说明置信度的来源（如"文档中有明确的字段和数值对应"vs"通过上下文推断，无直接对应字段"）。

置信度分数存储在 `extraction_results` 表中，前端在结果展示时以颜色编码（绿色高置信、黄色中等、红色低置信）直观呈现，并对低置信结果显示提示图标，引导用户重点审核。

**14.3.2 基于用户校验的准确率统计**

结合优化三（提取结果持久化）中的人工校验功能，建立统计分析能力：
- 按标签统计各标签的平均置信度和人工确认率
- 按模型统计不同 LLM 在各标签类型上的准确率
- 按文档类型统计不同格式文档的提取效果

在系统仪表板（或设置页）展示这些统计图表，帮助用户了解系统在自己业务数据上的实际表现。

**14.3.3 A/B 参数对比实验**

在信息提取页面增加"实验模式"入口，允许用户对同一文档、同一标签组，设置两组不同的提取参数（如左边配置 top_k=3，右边配置 top_k=8），系统并行执行两次提取，结果在同一界面左右对比展示。

对比视图中，用户可以标记"哪边的结果更好"，这个偏好数据与参数配置一起存储，用于后续的参数推荐（基于历史偏好，推荐最适合该用户业务场景的参数组合）。

**14.3.4 Prompt 可视化与自定义**

在系统配置中增加"Prompt 管理"页面，展示系统当前使用的各类 Prompt 模板：
- 多标签提取系统 Prompt
- 单标签提取系统 Prompt
- RAG 增强问题生成 Prompt

每个 Prompt 模板可以：查看当前内容（只读模式）、切换到编辑模式修改内容、在修改前预览"使用修改后的 Prompt，对某个 chunk 和某个标签会产生什么样的 LLM 请求"、保存修改，支持版本历史，支持一键回滚到默认版本。

这一功能让有经验的用户可以针对中文业务场景、特定行业术语等优化 Prompt，而无需接触代码。

**14.3.5 提取效果基准测试**

提供一套简单的基准测试框架：允许用户创建一个"黄金数据集"（若干文档 + 每个文档各标签的正确答案），然后以不同参数组合自动化运行提取，并与黄金答案对比，计算每种配置的准确率、召回率等指标，帮助用户找到最优配置。

### 14.4 实施路径

1. 修改提取 Prompt，要求 LLM 返回置信度分数
2. 扩展 `extraction_results` 表，存储置信度数据
3. 前端提取结果展示中加入颜色编码的置信度指示
4. 实现 A/B 对比实验 UI
5. 实现 Prompt 管理页面（查看 + 编辑 + 版本历史）
6. 可选：实现基准测试框架

### 14.5 预期效果

专业用户和技术研究者获得了一套完整的提取质量评估工具，可以量化地比较不同配置的效果，系统性地优化 Prompt，并建立起针对自身业务场景的最优参数基线。这将 AnythingExtract 从"可以用"升级为"可以调优、可以信任"的专业工具。

## 15. 实施优先级与路线规划

### 15.1 三阶段路线图

**第一阶段：产品可用性（P0 优先级）**

目标：让普通用户能顺畅使用核心功能，完成从"演示工具"到"可用产品"的跨越。

包含优化建议：
- 优化一：提取方案模板系统（降低新手门槛，最直接的用户价值）
- 优化二：多标签批量提取与导出（企业用户的核心需求，决定产品能否投入实际工作）

完成这两项后，产品可以真正被非技术用户用于日常工作，而不仅仅是开发者的演示玩具。

**第二阶段：功能完整性（P1 优先级）**

目标：补全规划中但未实现的功能模块，提升系统技术深度。

包含优化建议：
- 优化三：提取结果持久化（数据资产沉淀，支撑后续所有分析功能）
- 优化四：智能分块策略（检索质量基础提升，影响所有提取场景）
- 优化五：Rerank 精排落地（完成 Stage 4，达到规划的技术目标）
- 优化六：知识问答模块（填补空壳页面，增加产品完整性）

**第三阶段：专业化与工程化（P2/P3 优先级）**

目标：提升系统成熟度，扩大用户覆盖，建立开源项目生态。

包含优化建议：
- 优化七：多 Provider 支持（扩大目标用户群）
- 优化八：文档预览与溯源（专业审核场景）
- 优化九：向量检索性能优化（规模化准备）
- 优化十：Docker 部署（开源工程化）
- 优化十一：流式提取与进度（体验打磨）
- 优化十二：质量评估系统（专业化工具）

### 15.2 各优化建议之间的依赖关系

```
优化一（模板）→ 优化二（批量提取）  // 批量提取与方案模板结合更自然
优化三（结果持久化）→ 优化十二（质量评估）  // 质量评估需要历史结果数据
优化四（分块策略）→ 优化五（Rerank）  // 分块质量是 Rerank 效果的前提
优化二（批量提取）→ 优化十一（实时进度）  // 批量进度是流式体验的重要场景
优化八（文档预览）→ 优化三（溯源链接）  // 溯源定位功能需要文档预览支持
优化七（多Provider）→ 优化六（QA模块）  // QA 模块用云端大模型效果更好
```

其他优化建议之间无强依赖，可并行进行。

### 15.3 技术风险提示

**风险一：LanceDB 数据迁移**  
优化九中知识库分表改造需要迁移现有向量数据，存在数据丢失风险。实施前需编写完整的迁移脚本并在测试环境充分验证，同时提供数据备份和回滚方案。

**风险二：Embedding 模型切换导致维度不兼容**  
优化七中更换 Embedding Provider 或模型后，新旧向量维度不同，旧向量无法参与检索。需要在切换前提示用户"所有文档需要重新向量化"，并提供一键重建功能。

**风险三：批量提取的资源占用**  
优化二的批量任务在处理大量文档时会持续占用系统资源（Ollama GPU/CPU），可能影响同期进行的其他操作。需要在批量任务中设置适当的资源限制和并发上限，并在 UI 中明确提示用户。

**风险四：Prompt 自定义的滥用**  
优化十二允许用户修改系统 Prompt，不当修改可能导致 JSON 解析失败或提取逻辑异常。需要提供 Prompt 沙盒测试功能，并保留一键回滚到默认版本的安全选项。
